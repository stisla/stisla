## Working in sessions

Every migration step is its own session ‚Äî long-running work doesn't fit in one conversation. Each session should be self-bootstrapping from this file + V3.md + git history. No conversation context required.

### Maintainer: how to start a session

Open Claude Code in this repo and paste:

> "Let's continue the v3 migration. Read V3.md + MIGRATION.md, check git, and tell me what step we're on and what to do next."

That's it. Don't pre-load assumptions; the agent re-derives state from the docs + git.

### Agent: what to do on cold start

1. **Read V3.md.** The spec. Agent workflow rules at the top still apply ‚Äî no writes until the maintainer says "go".
2. **Read MIGRATION.md.** Status (top) first, then the ¬ß3 step relevant to today's work, then the ¬ß4 disposition row if working on a component. **Don't read MIGRATION-LOG.md by default** ‚Äî `grep` it only when you need precedent from a completed session (e.g., naming conventions, var-grouping decisions, follow-up callouts).
3. **Verify against git.** Don't trust Status blindly. Run `git log --oneline -20`, `git status`, `git tag --list 'v3-*'` to see what actually shipped. If the doc and git disagree, the doc is stale ‚Äî flag it and propose an update.
4. **Identify next work.** ¬ß7 running checklist is the authoritative source for "what step are we on". Within a step, ¬ß3 has the sub-bullets; ¬ß4 has the per-component table.
5. **Confirm with the maintainer** what to do before touching code. Default proposal: "next unchecked box in ¬ß7" ‚Äî but the maintainer may want to skip ahead, revisit, or branch off.
6. **At session end:** add one new line to the Status section (top of file). Format: `ISO date ‚Äî what changed ‚Äî blockers if any`. Commit as part of the session's final commit, or as a standalone `docs(v3): update migration status`. If the verbose Status block has grown past ~3 entries, move the oldest to the **top** of MIGRATION-LOG.md in the same commit so the operational doc stays slim.

### When in doubt

- Spec questions (token names, naming rules, what's in/out of scope) ‚Üí V3.md.
- Migration questions (order, per-component action, comparison workflow) ‚Üí MIGRATION.md.
- Precedent from a completed session (decisions, conventions, naming maps) ‚Üí `grep MIGRATION-LOG.md`. Don't read it whole; it's the archive.
- "Is this thing actually built?" ‚Üí git, not the docs.
