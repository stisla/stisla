# Stisla v3

> Single source of truth for v3 design, scope, and progress.
> Conversations get lost; this file doesn't. Update as decisions are made.

---

## 1. Vision

Stisla v3 is a full rewrite. It is a CSS-first admin framework built on Bootstrap 5.3, designed to be:

- **Theme-able via CSS variables** — rebrand without touching Sass.
- **Framework-agnostic** — vanilla JS that React/Vue/Svelte can wrap trivially.
- **Showcased through the site itself** — `stisla.dev` IS the docs. Component pages double as live demos with view-source; template pages double as marketing and as downloadable starters.
- **Honest about scope** — minimal core, no abandoned dependencies, no half-finished features.

What v3 is NOT:

- Not an upgrade from v2 (clean break, no migration path).
- Not a kitchen sink (no rich-text editor, no datatables, no chart library bundled).
- Not jQuery-based.

## 2. Decisions (locked)

| Area | Decision | Why |
| --- | --- | --- |
| Base framework | Bootstrap 5.3, with BS6 forward-compatibility in mind | BS5.3 already exposes most CSS vars we need; BS6 not ready |
| Theming model | Stisla primitives + BS5 vars. No Stisla semantic/component aliases. | One naming API (BS5's `--bs-*`), no parallel scheme to learn. Add `--stisla-*` only for concepts BS5 doesn't model. |
| Distribution | npm + CDN | Standard for CSS frameworks; covers both ecosystems |
| Repo shape | Single tree, no monorepo tools | Simpler; framework and site share build deps |
| Site templating | Nunjucks (templates are also the deliverables) | Clean HTML output; same source for live demo + downloadable template |
| Build tooling | Vite (site + HMR), Sass CLI + PostCSS (CSS bundles), Node scripts for Nunjucks | Gulp drops out |
| JS | Vanilla, class-based, `destroy()` mandatory, custom events | Framework-agnostic by contract |
| Browser support | Last 2 versions Chrome/Edge/Firefox, Safari 15.4+ | Buys `color-mix()`, `:has()`, `@layer`, container queries |
| v2 compatibility | None. Clean break. | Full rewrite, not worth migration cost |
| v2 originals | Cut from 3.0. Reappear as recipe pages, or as components in 3.1+ if utilities can't do it. | Most were card + utility compositions |
| Home | `stisla.dev` | New domain; v2 stays at getstisla.com as archive |

## 3. Architecture

### 3.1 Token layers

Two layers. Stisla owns the primitives. BS5 owns everything semantic and component-level.

```css
/* Stisla primitives — BS5 has no equivalent */
--stisla-blue-50 ... --stisla-blue-950
--stisla-gray-50 ... --stisla-gray-950
/* (... only scales BS5 doesn't tune the way we want) */

/* Default seeding — wire BS5 vars at our chosen primitive values */
--bs-primary: var(--stisla-blue-500);
--bs-body-bg: var(--stisla-gray-50);
--bs-body-color: var(--stisla-gray-900);
/* ...only where Stisla differs from BS5 defaults */
```

**Customization API = BS5's own vars.** Users override `--bs-primary`, `--bs-btn-bg`, `--bs-card-spacer-y`, etc. — Bootstrap's existing API, unchanged. No parallel `--stisla-*` alias to learn.

`--stisla-*` semantic tokens appear only for concepts BS5 doesn't model (e.g. a layered surface system, future Stisla originals). Added when needed, not preemptively.

**Dark mode** uses BS5's built-in `[data-bs-theme="dark"]`. Primitive overrides under that selector handle any palette tuning we want for dark.

### 3.2 JS contract

Three rules for every Stisla JS component:

1. **Class-based** — `new Stisla.X(el, opts)`.
2. **Destroy method** — every instance has `.destroy()`. Non-negotiable; this is what makes React unmount work.
3. **Custom events** — emit DOM events (`stisla:dropdown:open`, etc.), not callbacks.

Plus a declarative path for plain-HTML users:

```html
<div data-stisla-dropdown>...</div>
<script>Stisla.init();</script>
```

BS5 vanilla JS is re-exported under `Stisla.bs.*` so users get one entry point.

### 3.3 Repo layout

```
/src/
  scss/
    tokens/         CSS var layers + bridge
    core/           BS5 customization (Sass overrides)
    bundles/        bundle entry points
  js/
    core/           BS5 vanilla re-exports
    components/     Stisla components (post-3.0 mostly)
    index.js
  site/
    pages/          /buttons, /cards, ... (component demos)
    templates/      dashboard-1, auth, ... (full templates)
    layouts/
    partials/
/dist/              built CSS+JS bundles (npm + CDN)
/site-dist/         built static site (stisla.dev)
vite.config.js
```

### 3.4 Bundles

| File | Contains | Audience |
| --- | --- | --- |
| `stisla-tokens.css` | CSS vars only, no styles | Headless / power themers |
| `stisla-core.css` | Tokens + restyled BS5 | Most users |
| `stisla-components.css` | Tokens + Stisla originals | 3.1+ only — defer until originals exist |
| `stisla-full.css` | Everything | Default CDN bundle |

JS bundles: `stisla.js` (UMD), `stisla.esm.js` (ESM, tree-shakable). Decide if separate core/components JS bundles are warranted in Phase 3.

## 4. Phases

Each phase has explicit "done when" criteria. No vague endings.

### Phase 0 — Foundation
- [x] `v3` branch cut
- [x] `V3.md` committed (this file)
- [x] Repo dirs scaffolded per §3.3
- [x] Vite + Sass + Nunjucks + PostCSS build pipeline runs `npm run dev` and `npm run build`
- [x] `hello world` page renders with hot reload

### Phase 1 — Tokens + theme
- [ ] Primitive scales drafted (only the scales BS5 doesn't tune the way we want)
- [ ] BS5 vars seeded from primitives where Stisla's defaults differ
- [ ] `stisla-tokens.css` builds
- [ ] Dark mode works via BS5's `[data-bs-theme="dark"]` with primitive overrides
- [ ] Brand recolor demo: change `--bs-primary` (or a primitive), sample page rebrands without Sass rebuild

### Phase 2 — Core CSS
- [ ] BS5 restyled: buttons, forms, cards, modals, dropdowns, nav, navbar, alerts, badges, tables, pagination, tabs, accordion, toasts, offcanvas
- [ ] Dark mode verified for each
- [ ] `stisla-core.css` and `stisla-full.css` build
- [ ] Visual parity check against BS5 base (we keep BS5 semantics, we restyle)

### Phase 3 — JS contract
- [ ] `Stisla.init()` declarative scanner
- [ ] One reference component fully implements the contract (class + destroy + events)
- [ ] BS5 vanilla re-exports under `Stisla.bs.*`
- [ ] `stisla.js` (UMD) + `stisla.esm.js` build

### Phase 4 — Templates
- [ ] `dashboard-1` built using only Phase 1+2+3 output
- [ ] `auth` (login + register) built same way
- [ ] Per-template download zip generated at build time

### Phase 5 — Site (stisla.dev)
- [ ] Landing page
- [ ] Component pages for everything in Phase 2 with view-source panel
- [ ] Template pages (Phase 4) with download button
- [ ] Foundational docs: installation, theming, JS API, browser support
- [ ] Patterns pages (the old v2 originals reborn as recipes)

### Phase 6 — Release
- [ ] npm publish `@stisla/css` and `@stisla/js` (package names TBD)
- [ ] CDN URLs verified (jsdelivr / unpkg)
- [ ] `stisla.dev` live with DNS
- [ ] git tag `v3.0.0`
- [ ] CHANGELOG / announcement

## 5. Open questions

- [ ] Should `stisla-components.css` exist in 3.0 if no originals ship? (Probably defer the bundle until 3.1.)
- [ ] Adopt CSS `@layer` ordering? Likely yes: `@layer reset, tokens, bootstrap, stisla, utilities`.
- [ ] Sass entry point naming — `_index.scss` vs explicit files?
- [ ] Per-template zip: build-time script or runtime endpoint? (Static site → build script.)
- [ ] Do we ship a CLI scaffolder in 3.0 or wait for demand? (Lean: defer.)
- [ ] Final npm package names — `stisla`, `@stisla/css` + `@stisla/js`, or single `stisla` with subpaths?
- [ ] Keep v2 archive at `getstisla.com`, or 301 to `stisla.dev/v2`?

## 6. Out of scope (3.0)

These get recipe pages on the site pointing users to "bring your own". They do NOT ship in the CSS or JS bundles.

- Rich-text editor (was summernote in v2)
- Datatables
- Chart library (chart.js etc.)
- File upload widget (was dropzone)
- Date/time pickers
- Maps (was gmaps/jqvmap)
- Calendar (was fullcalendar)
- RTL stylesheet (defer to 3.1)
- i18n examples
- Framework adapter packages (React/Vue/Svelte wrappers)
- Migration tooling from v2
- v2 original components (reborn as recipes; promote to real components in 3.1+ only when utilities can't do it)
