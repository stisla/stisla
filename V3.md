## Agent workflow rules (read before doing anything)

You are an AI agent working on Stisla v3. Before you do **anything**:

1. **Read this entire V3.md.** It survives across sessions; the conversation doesn't.
2. **Before suggesting any Bootstrap customization, read the relevant docs page** at <https://getbootstrap.com/docs/5.3/customize/> — overview, sass, options, color, color-modes, components, css-variables, optimize. Don't guess at what BS5 has or how it's named; the maintainer will catch it.
3. **When docs are ambiguous, grep `node_modules/bootstrap/scss`** for the actual implementation. The source is ground truth.
4. **Plan in chat first.** Lay out what you intend to write, the exact files, the exact override values, the import order. Wait for the maintainer's "go". **Do not write files until the plan is approved.**
5. **Follow BS5 conventions; don't invent.** Zero `--stisla-*` runtime vars in v3. Customization flows through `--bs-*` (root) and BS5's per-component vars. Don't add loops, layers, or aliases BS5 doesn't already ship.
6. **One Sass file = one job.** See `src/scss/tokens/` for the Phase 1 pattern.
7. **Never push or merge** without explicit maintainer approval.

Skipping these wastes a round trip. They're not aspirational.

## 1. Vision

Stisla v3 is a full rewrite. CSS-first admin framework built on Bootstrap 5.3:

- **Theme-able via CSS variables** — rebrand by overriding `--bs-*` at runtime (with caveats; see §3.1).
- **Framework-agnostic** — vanilla JS that React/Vue/Svelte can wrap trivially.
- **Showcased through the site itself** — `stisla.dev` IS the docs. Component pages double as live demos with view-source; template pages double as marketing and as downloadable starters.
- **Honest about scope** — minimal core, no abandoned dependencies, no half-finished features.

What v3 is NOT:

- Not an upgrade from v2 (clean break, no migration path).
- Not a kitchen sink (no rich-text editor, no datatables, no chart library bundled).
- Not jQuery-based.

## 3. Architecture

### 3.1 Token layers

**One layer at runtime: Bootstrap's `--bs-*` CSS variables.** Stisla's contribution lives at Sass build time — we override BS5's `!default` Sass vars with our hand-picked palette and BS5 emits the correct `:root` vars from there.

The override sequence follows Bootstrap's "Option B" customization workflow exactly:

```
functions → our Sass overrides → variables → variables-dark → our map overrides
         → maps → mixins → utilities → root → reboot → ... → utilities/api
```

**No `--stisla-*` runtime vars.** Bootstrap already names what we need:

- `--bs-primary`, `--bs-blue`, `--bs-gray-100..900`, etc. for colors
- `--bs-body-bg`, `--bs-body-color`, `--bs-emphasis-color`, `--bs-secondary-color`, `--bs-tertiary-color` for body/text
- `--bs-border-color` for borders
- The theme-color triplet (`--bs-primary-bg-subtle`, `--bs-primary-text-emphasis`, `--bs-primary-border-subtle`) for tinted callouts/alerts/badges
- Per-component vars (`--bs-card-bg`, `--bs-btn-hover-bg`, etc.) for component-level overrides

Hover / active / disabled are seeded from our palette via existing BS5 component vars (e.g., `$nav-link-disabled-color: $gray-300`). No new var categories.

**Dark mode** uses BS5's built-in `[data-bs-theme="dark"]`. BS5 auto-generates dark-mode triplet variants via `shade-color()` / `tint-color()` from our base values.

**Runtime override caveat (important).** BS5 components (buttons, alerts, badges, cards, etc.) compile their colors at Sass build time from `$primary`, `$danger`, etc. Overriding `--bs-primary` at runtime does NOT recolor them. What DOES respond at runtime:

- Utility classes that use `var(--bs-primary-rgb)`: `.bg-primary`, `.text-bg-primary`, `.text-primary`, `.border-primary`
- Anything you write that explicitly references `var(--bs-primary)`

For component recoloring, users rebuild from Sass. This is a BS5 architectural limit; document it in the Phase 5 site docs.

### 3.2 JS contract

Three rules for every Stisla JS component:

1. **Class-based** — `new Stisla.X(el, opts)`.
2. **Destroy method** — every instance has `.destroy()`. Non-negotiable; this is what makes React unmount work.
3. **Custom events** — emit DOM events (`stisla:dropdown:open`, etc.), not callbacks.

Plus a declarative path for plain-HTML users:

```html
<div data-stisla-dropdown>...</div>
<script>Stisla.init();</script>
```

BS5 vanilla JS is re-exported under `Stisla.bs.*` so users get one entry point.

### 3.3 Repo layout

```
/src/
  scss/
    tokens/        ← Sass-time tokens (_colors, _overrides, _maps)
    bundles/       ← bundle entry points (stisla-full.scss)
  js/
    core/          ← BS5 vanilla re-exports (Phase 3)
    components/    ← Stisla components (3.1+)
    index.js
  site/
    pages/         ← /buttons, /cards, ... (component demos)
    templates/     ← dashboard-1, auth, ... (full templates)
    layouts/
    partials/
/tools/            ← build scripts (vite-plugin-nunjucks.mjs, render-site.mjs)
/dist/             ← built CSS+JS bundles for npm + CDN (Phase 6)
/site-dist/        ← built static site for stisla.dev
vite.config.mjs
postcss.config.mjs
```

### 3.4 Bundles

**3.0 ships one CSS bundle:** `stisla-full.css` (~231 KB unminified, ~31 KB gzipped) and one JS bundle: `stisla.js`.

Bundle splitting (`stisla-tokens.css` for headless themers, `stisla-core.css` for slim builds without Stisla originals, `stisla-components.css` for Stisla-only bolt-on use) is post-3.0 scope. Add when there's a real ask.

## 6. Out of scope (3.0)

These get recipe pages on the site pointing users to "bring your own". They do NOT ship in the CSS or JS bundles.

- Rich-text editor (was summernote in v2)
- Datatables
- Chart library (chart.js etc.)
- File upload widget (was dropzone)
- Date/time pickers
- Maps (was gmaps/jqvmap)
- Calendar (was fullcalendar)
- RTL stylesheet (defer to 3.1)
- i18n examples
- Framework adapter packages (React/Vue/Svelte wrappers)
- Migration tooling from v2
- v2 original components (reborn as recipes; promote to real components in 3.1+ only when utilities can't do it)
