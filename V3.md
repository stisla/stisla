## Agent workflow rules (read before doing anything)

You are an AI agent working on Stisla v3. Before you do **anything**:

1. **Read this entire V3.md.** It survives across sessions; the conversation doesn't.
2. **Stisla is a design spec, not "a Bootstrap skin".** BS5 SCSS is gone — we fork BS5's grid + breakpoint mixins (MIT-credited under `LICENSES/`) and ship our own everything else. No `@use "bootstrap/..."` anywhere. No BS5 JS bundle. See §3.1 + §3.11.
3. **Runtime surface is `--st-*` only.** 30 OKLCH tokens. No `--bs-*` references in component CSS. Every bg-providing token has a paired `-foreground`. State derivation is `color-mix(in oklch, …)` over runtime `--st-*` vars.
4. **No SASS color math.** No `tint-color`, `shade-color`, `shift-color`, `lighten`, `darken`, `rgba($var, X)`. If you need a hover state, write `color-mix(in oklch, var(--st-X) 88%, black)`.
5. **No spacing ramp.** Component padding is a literal × `var(--st-density)` (§3.4). No `--st-space-*` tokens.
6. **BEM + state hooks.** `.block`, `.block__element`, `.block--modifier`. Runtime state: `[data-state="open|active"]` for Radix-mapped concepts, `.is-*` for Stisla-original (`.is-loading`, `.is-collapsed`). See §3.6.
7. **One Sass file per component.**
8. **Plan in chat first.** Lay out the files, the class names, the token usage. Wait for the maintainer's "go". Do not write files until the plan is approved.
9. **Never push or merge** without explicit maintainer approval.

Skipping these wastes a round trip. They are not aspirational.

## 1. Vision

Stisla is a design specification. v3 ships the spec and the vanilla implementation. Other framework implementations (React + Base UI/Radix, Vue + Reka UI, Svelte + bits-ui) conform to the same spec; they're future work, not adapters.

- **The spec defines "what".** Anatomy, parts, states, behavior, a11y requirements, token surface, visual identity. Framework-agnostic.
- **Implementations define "how".** Vanilla uses our own JS (Floating UI, focus-trap, Embla). React uses Radix/Base UI primitives + Stisla CSS. Each impl uses what's idiomatic in its environment.
- **CSS is the universal layer.** BEM classes + OKLCH tokens are shared across every implementation. The behavior layer is per-environment.
- **True runtime theming.** One root token override recolors every component. Brutalist mode (radius 0, solid shadow) is a one-line override. Density scale is a one-line override.
- **Small token surface.** 30 `--st-*` tokens at the root, named by purpose, not by color slot. Every bg-providing token has a paired `-foreground`.
- **BEM naming + Radix-aligned state attrs.** `.btn`, `.btn--primary`. Runtime state uses `[data-state="open"]` for Radix-mapped concepts, `.is-*` for Stisla-original states (`.is-loading`, `.is-collapsed`).
- **Showcased through the site itself.** `stisla.dev` IS the docs. Component pages double as live demos with view-source; template pages double as marketing and as downloadable starters.
- **Honest about scope.** Minimal core, no abandoned dependencies, no half-finished features.

What v3 is NOT:

- Not an upgrade from v2 (clean break, no migration path).
- Not a kitchen sink (no rich-text editor, no datatables, no chart library bundled).
- Not jQuery-based.
- Not a Bootstrap skin. BS5 SCSS is gone entirely; we fork its grid + breakpoint mixins (MIT-credited) and compose vanilla JS primitives ourselves.

## 3. Architecture

### 3.1 Foundation layer

Zero `@use "bootstrap/..."` in SCSS. BS5 SCSS dependencies are gone — we tried importing-but-not-really and the constant friction with BS variables, mixins, and `$theme-colors` was the cost. Forking the pieces we want and shipping our own everything else removes the friction.

| Piece | Source | Notes |
| --- | --- | --- |
| Reset | `modern-normalize` (vendored) + `foundation/_reboot.scss` (Preflight-style: `* { margin: 0; padding: 0; border: 0 solid }`; `h1`–`h6` inherit size/weight; `<a>` inherits color/decoration; `b`/`strong`/`em`/`i`/`small` lose visual treatment; `ul`/`ol` lose markers; form controls inherit font; replaced elements `display: block` + `max-width: 100%`. Keeps v3 chrome: `<mark>` highlight, `<abbr>` dotted underline, `<hr>` via `--st-border`, `:focus-visible` at `--st-ring`). | One global pass intercepts HTML-element defaults before they leak into components. |
| Container + grid + row/col | Forked from `bootstrap/scss/_containers.scss` + `bootstrap/scss/_grid.scss` + `bootstrap/scss/mixins/_grid.scss`. Stored under `foundation/_grid.scss`. Strip `--bs-*` emits, RTL margin flips, deprecated `.no-gutters`, `$enable-cssgrid` branch. Rename gutter vars to `--st-grid-gutter-x/y`. | Familiar BS-like flex grid; our `--st-*` namespace. |
| Breakpoint mixins | Forked from `bootstrap/scss/mixins/_breakpoints.scss` into `foundation/_breakpoints.scss`. Rename `media-breakpoint-up` → `media-up`, etc. | Breakpoints are the one place Sass survives — media queries can't read CSS vars. |
| `color-mode` mixin | Rewritten (~3 lines) in `foundation/_mixins.scss`. | Trivial; not worth forking. |

License compliance: each forked file carries a header comment crediting `Bootstrap 5.3 (MIT) — https://github.com/twbs/bootstrap` with the original path. A `LICENSES/bootstrap-MIT.txt` lives at repo root with the full BS5 license text. README NOTICE line.

**JS layer (vanilla impl).** No `bootstrap.bundle.js`. Composed primitives instead:

| Need | Library | Size gz |
| --- | --- | --- |
| Positioning (tooltip, popover, dropdown) | `@floating-ui/dom` | ~6kb |
| Focus trap (modal, drawer, dropdown menu) | `focus-trap` + `tabbable` | ~3kb |
| Carousel | `embla-carousel` | ~6kb |
| Modal/drawer/dropdown/tabs/accordion/toast/tooltip/popover behavior | Stisla code (~600 lines total) | n/a |

Total ~25–30kb gz — comparable to BS5 bundle, with better output (animated state transitions, FLIP indicators, presence animations). See §3.7 for the vanilla JS contract. Other framework implementations use their ecosystem's primitive library (Radix/Base UI for React, Reka UI for Vue, bits-ui for Svelte) — see §3.11.

**Visual treatment — three layers, no gaps.** With Preflight in place, the rule for where any styling lives is:

1. **Preflight** (`foundation/_reboot.scss`) handles the global reset. Every HTML element starts blank.
2. **`.prose`** (`components/_prose.scss`) handles long-form restoration. Inside it, `<strong>`/`<em>`/`<small>`/`<a>`/`<ul>`/`<ol>`/`<blockquote>`/`<table>`/`<img>` get reading-context defaults. Customize via 5 `--st-prose-*` vars and the cascade.
3. **App UI** composes from primitives: utilities (`.fw-*`, `.fs-*`, `.text-*`, `.link`), patterns (`.h1`–`.h6`, `.lead`, `.display-*`), and components (`.btn`, `.card`, `.alert`, ...).

No utilities live in the gap between those three. `.blockquote` doesn't ship as a utility (long-form → `.prose`). `.list-unstyled` doesn't ship (Preflight already strips bullets). `.list-inline` doesn't ship (BS5 layout shortcut → flex + gap or a component). If you're tempted to add a utility that's "the way the element looked in BS5", check which of the three layers actually owns that styling.

### 3.2 Token surface (`--st-*`)

The entire public customization API. 30 tokens. All values authored as OKLCH literals in `tokens/_theme.scss` — the only file in the codebase with literal color values. No Sass color ramps anywhere. No runtime `calc()` chain — every token is a literal OKLCH value.

**Pairing rule.** Every token usable as a `background` has a paired `-foreground`. Without this, hardcoding `color: #fff` on `.btn--primary` breaks the moment someone overrides `--st-primary` with a light hue.

```scss
// Intent (10) — 5 pairs
--st-primary
--st-primary-foreground
--st-success
--st-success-foreground
--st-warning
--st-warning-foreground       // dark — yellow needs dark text in both themes
--st-danger
--st-danger-foreground
--st-info
--st-info-foreground

// Surface tier (7)
--st-background               // page bg
--st-foreground               // body text
--st-surface                  // cards, modals, popovers, dropdowns
--st-surface-2                // alt rows, card footers, accordion headers
--st-surface-3                // deepest tier
--st-border
--st-muted-foreground         // muted body text (was --st-muted; renamed for consistency)

// Interactional (6) — 3 pairs
--st-neutral                  // rest bg for filled-neutral elements (btn--neutral, kbd, badge default, input-group addons)
--st-neutral-foreground
--st-accent                   // hover bg (neutral)
--st-accent-foreground
--st-highlight                // selected / current bg (soft primary tint)
--st-highlight-foreground

// Focus (1)
--st-ring                     // focus outline color; defaults to --st-primary

// Geometry (3)
--st-radius                   // 0 = brutalist, 0.75rem = default, 1rem = soft
--st-shadow                   // "4px 4px 0 black" = neo-brutalist; default = soft ambient
--st-density                  // 1 default, 0.875 compact, 1.125 comfortable

// Type (2)
--st-font-sans
--st-font-mono                // code, kbd, pre, tabular-figures contexts
```

**Neutral, accent, highlight.** Three distinct interactional semantics, one per state in the rest → hover → selected progression:

| Use | Rest fill | Hover | Selected/current |
|---|---|---|---|
| `.btn--neutral` | `--st-neutral` | `--st-accent` | n/a |
| Sidebar item | (transparent) | `--st-accent` | `--st-highlight` (current page) |
| Dropdown item | (transparent) | `--st-accent` | `--st-highlight` (checked) |
| Table row | (transparent) | `--st-accent` | `--st-highlight` (multi-select) |
| Tab | (transparent) | `--st-accent` | `--st-highlight` (active tab) |
| Accordion header | (transparent) | `--st-accent` | `--st-highlight` (expanded) |
| `.kbd`, `.badge` (no intent), input-group addon | `--st-neutral` | n/a | n/a |

Neutral is a persistent fill weight for static neutral surfaces. Accent is transient hover feedback over those (and over otherwise-transparent) surfaces. Highlight is persistent selected state. Three concepts, three names.

**Retinting neutrals.** Default neutrals ship as pure gray (`oklch(L 0 0)`). To shift a subtree warm, cool, or any direction, override the neutral tokens in a parent scope — inline style, a class, a `[data-theme]` block, anything. Components inside read tokens via `var()` and inherit the new values automatically. Each retint scope spells out OKLCH literals for the affected tokens (commonly the 9 neutrals: `background`, `foreground`, `surface`, `surface-2`, `surface-3`, `border`, `muted-foreground`, `neutral`, `accent`). No runtime math — the math happens once at authoring time (or in a future theme generator tool) and ships as the numbers. Intents stay vivid regardless of retint scope.

**No spacing ramp.** Padding is component-local. See §3.4.

Derived tokens (`--st-radius-sm`, `--st-radius-lg`) compute via `calc()` as sibling vars where useful. They are not part of the customization story.

### 3.3 Color model

**Three systems, kept separate:**

**Intent colors.** Genuinely colorful. One color per intent (`--st-primary`, `--st-success`, etc.) with a paired `-foreground`. All hover/active/disabled states derive at runtime via `color-mix(in oklch, …)`.

**Surface/neutral tier.** Pure gray (`oklch(L 0 0)`) by default. Layered tiers (`--st-surface`, `--st-surface-2`, `--st-surface-3`). To retint warm/cool/any-hue, override the affected tokens in a parent scope — components inside inherit via `var()`. No runtime math; new tints are new OKLCH literals.

**Interactional trio.** `--st-neutral` is the rest fill for neutral surfaces (filled neutral button, kbd, default badge, input-group addons). `--st-accent` is the transient hover bg over neutral or transparent surfaces. `--st-highlight` is the persistent-state selected bg (soft primary tint by default). All three paired with foregrounds.

**No color ramps.** No `_colors.scss`, no `$blue-100..900`. Discipline argument: ramps make components lazy ("shoot, I'll just grab gray-400"). If a component needs a value, it's authored as an OKLCH literal in the component file (small, scoped) or, if it belongs in the design language, added to `_theme.scss` as a named token.

**OKLCH for mixing.** All `color-mix` happens `in oklch` for perceptually correct hover/active/subtle-bg derivations. The sRGB equivalent muddies on mid-saturation hues — the very ones you'd pick for brand colors.

**"Secondary" cleanup.** Three concepts, three names:

| Old (BS5 convention) | v3 | Meaning |
| --- | --- | --- |
| `.btn-secondary` | `.btn--neutral` | Quieter button next to primary. Neutral-derived. |
| `.text-secondary` | `.text-muted` | Muted body text — reads `--st-muted-foreground`. |
| `.alert-secondary` | `.alert` (no modifier) | Neutral default. Intent modifier opt-in. |
| `.bg-secondary` | dropped | Use `.bg-surface-2` or `.bg-muted` per purpose. |
| `.border-secondary` | dropped | Use `.border` (neutral) or intent border utility. |

Intent words stay (`--primary`, `--success`, `--warning`, `--danger`, `--info`). The neutral word is `--neutral` for buttons; `muted` for text; "no modifier" for neutral alert, badge, etc.

### 3.4 Component-scoped fallback pattern

Every component reads a component-scoped var that falls back to the global token:

```css
.btn   { border-radius: var(--st-btn-radius, var(--st-radius)); }
.card  { border-radius: var(--st-card-radius, var(--st-radius)); }
.input { border-radius: var(--st-input-radius, var(--st-radius)); }
```

Global override: set `--st-radius` once at `:root`. Per-component override: set `--st-btn-radius`. Same pattern for density and shadow.

**`--st-radius` applies to the outer root only.** Inner elements (nested items, framed list rows, accordion items inside an accordion frame, sidebar items inside a padded sidebar) must compute their own radius so the inner shape stays concentric with the outer shape. The rule:

```
inner radius = outer radius - outer padding
```

In CSS:

```css
.accordion {
  --st-accordion-radius: var(--st-radius);
  --st-accordion-padding: 0.25rem;
  border-radius: var(--st-accordion-radius);
  padding: var(--st-accordion-padding);
}
.accordion__item {
  border-radius: calc(var(--st-accordion-radius) - var(--st-accordion-padding));
}
```

If the inner element keeps the same radius as the outer, the inner corner sits inside the outer corner and reads as off-center. If the inner radius is the difference, the two arcs share a center and the frame reads clean. Apply this anywhere a "framed" component nests rounded items inside a padded root.

**Components that opt out of `--st-radius`** (shape is semantic, not stylistic):

- `.badge` (pill is intentional)
- `.avatar` (circle is intentional)
- `.spinner` (must stay round)
- `.form-check-input` checkbox / radio (square / circle is semantic)
- `.slider` track + thumb (pill + circle are semantic)
- `.progress` track (pill is intentional for a progress strip)
- `.btn--icon-round` (opt-in round square button)

Listed on the customization page so consumers know which knobs do nothing.

**Spacing — no ramp.** No `--st-space-*` tokens at `:root`. Padding is a component-local literal, wrapped in `calc(X * var(--st-density))` so the one density knob retunes the whole system:

```scss
.btn {
  --st-btn-padding-y: calc(0.5rem  * var(--st-density));   //  8 × density
  --st-btn-padding-x: calc(0.75rem * var(--st-density));   // 12 × density
  padding: var(--st-btn-padding-y) var(--st-btn-padding-x);
}

.alert {
  --st-alert-padding: calc(1rem * var(--st-density));      // 16 × density
  padding: var(--st-alert-padding);
}

.card {
  --st-card-padding: calc(1.25rem * var(--st-density));    // 20 × density
  padding: var(--st-card-padding);
}

.kbd {
  --st-kbd-padding-y: calc(0.125rem * var(--st-density));  //  2 × density
  --st-kbd-padding-x: calc(0.375rem * var(--st-density));  //  6 × density
  padding: var(--st-kbd-padding-y) var(--st-kbd-padding-x);
}
```

The 6px `.kbd` outlier isn't special — it's just a literal next to the component. Discipline argument (same as colors): a global ramp encourages "I'll grab `--st-space-3`" laziness and rots the +4 visual rhythm. Without a ramp, components state their value at the point of use, where it's reviewable and deletable.

**Design guideline (not enforced).** Prefer 4px-step increments: 6, 8, 12, 16, 20, 24, 28. Pick the value that fits the hierarchy. Reviewer pushes back on a 13px PR; not a runtime constraint.

**Density is the one global lever.** `[data-density="compact"] { --st-density: 0.875 }` shrinks every component proportionally. No per-component overrides to maintain.

On **height-aware components** (`.btn`, `.form-control` single-line, `.icon-box`, `.spinner`), density multiplies the component's hard `height` value, not just padding. The pattern is `height: calc(var(--component-height) * var(--st-density))` with `padding-block: 0` for single-line shapes (native vertical centering handles the content). Multi-line cases (`textarea`, `<select multiple>`, `.btn--wrap`) opt out via `height: auto` + a `min-height` floor + their own `padding-block`. The `--st-density` knob is then a real structural lever, not cosmetic — a `0.875` value genuinely produces a 31.5px-default form-control + 31.5px-default button, not a 36px shell with shrunk inner padding.

### 3.5 State derivation via color-mix

No per-state tokens. Every state derives from the base intent via `color-mix(in oklch, …)`:

```css
.btn--primary           { background: var(--st-primary); color: var(--st-primary-foreground); }
.btn--primary:hover     { background: color-mix(in oklch, var(--st-primary) 88%, black); }
.btn--primary:active    { background: color-mix(in oklch, var(--st-primary) 78%, black); }
.btn--primary:disabled  { background: color-mix(in oklch, var(--st-primary) 50%, transparent); }

.alert--primary {
  background:   color-mix(in oklch, var(--st-primary) 7%, transparent);
  border-color: color-mix(in oklch, var(--st-primary) 40%, transparent);
  color:        var(--st-foreground);
}

.badge--primary {
  background: color-mix(in oklch, var(--st-primary) 15%, transparent);
  color:      var(--st-primary);
}
```

Tune percentages per component family. Bake them into the component CSS. Do not expose as tokens.

**Why oklch and not srgb.** sRGB mixing produces muddy/gray midpoints on mid-saturation hues — the exact ones brands pick. OKLCH stays vivid through the mix because it interpolates in perceptual space. Visible difference on hover states for primary/info/danger.

### 3.6 BEM naming convention

```
.block                       // component
.block__element              // part of component
.block--modifier             // variant (static, set at render time)
.block[data-state="X"]       // runtime state — Radix-aligned concepts
.block.is-state              // runtime state — Stisla-original concepts
```

Examples:

```
.btn
.btn__icon
.btn--primary
.btn--lg
.btn--icon-only
.btn.is-loading                          // no Radix equivalent

.card
.card__header
.card__body
.card__footer
.card--flat

.alert
.alert__icon
.alert__heading
.alert__description
.alert__action
.alert--primary

.sidebar
.sidebar__header
.sidebar__content
.sidebar__footer
.sidebar__item
.sidebar__item[data-state="active"]       // current page
.sidebar.is-collapsed                     // no Radix equivalent

.modal
.modal__header
.modal__body
.modal[data-state="open"]                 // mirrors Radix Dialog

.dropdown-menu
.dropdown-menu__item
.dropdown-menu__item[data-highlighted]    // mirrors Radix Menu

.tabs__list[data-orientation="horizontal"]
.tabs__trigger[data-state="active"]
```

Rules:
- Lowercase, hyphen-separated inside block, element, and modifier segments.
- Element classes always under the block (`.card__header`, not `.header`).
- Variants use `--modifier`. Multiple modifiers compose flat on the root, not nested.
- **Runtime state — Radix-mapped:** use `[data-state="open|closed|active"]`, `[data-highlighted]`, `[data-disabled]`, `[data-orientation]` for any state that Radix/Base UI/Reka UI also expose. The same CSS then serves the vanilla impl (which toggles the attribute) and the React/Vue impls (which inherit the attribute from their primitive library) — no glue layer needed.
- **Runtime state — Stisla-original:** use `.is-*` for states without a Radix counterpart (`.is-loading` on `.btn`, `.is-collapsed` on `.sidebar`). The prefix signals mutability and lets generic JS helpers target `[class*="is-"]` patterns.
- **Native pseudos first.** `:disabled`, `:hover`, `:focus-visible` always take priority. data-attrs/`is-*` augment when a state is JS-driven and pseudo-classes don't apply.

| Component family | Hook to use |
|---|---|
| Modal, dropdown, popover, tooltip, accordion, collapsible | `[data-state="open\|closed"]` |
| Tabs, sidebar item (current page), pagination item, segmented control | `[data-state="active"]` |
| Menu item under keyboard nav | `[data-highlighted]` |
| Tab list, segmented control | `[data-orientation="horizontal\|vertical"]` |
| Button loading | `.is-loading` |
| Sidebar collapsed | `.is-collapsed` |

### 3.7 JS contract — vanilla implementation only

This section describes the vanilla JS implementation. Other framework implementations (React, Vue, Svelte) follow their environment's idioms — see §3.11.

Three rules for every Stisla JS component:

1. **Class-based.** `new Stisla.X(el, opts)`.
2. **Destroy method.** Every instance has `.destroy()`. Non-negotiable.
3. **Custom events.** Emit DOM events (`stisla:modal:open`, `stisla:dropdown:open`, etc.), not callbacks.
4. **State via data attributes.** Components toggle `data-state="open|closed|active"` on their root for Radix-aligned states; `.is-*` classes for Stisla-original states. Never both for the same concept.

Plus a declarative path for plain-HTML users:

```html
<div data-stisla-dropdown>...</div>
<script>Stisla.init();</script>
```

**Building blocks.** The vanilla impl composes well-known primitives. Stisla code stays small (~600 lines) and focuses on coordination + event dispatch:

- `@floating-ui/dom` — tooltip, popover, dropdown positioning
- `focus-trap` + `tabbable` — modal, drawer, dropdown menu focus management
- `inert` attribute — modal/drawer backdrop (AT-hide rest of page + scroll lock)
- `embla-carousel` — carousel
- Native `<details>` — collapsible (with thin JS for animated height transition)

### 3.8 Dark mode

Two selector conventions, both supported out of the box:

- **`[data-theme="dark"]`** attribute, typically on `<html>`. Idiomatic for vanilla / Rails / Django / Laravel consumers.
- **`.dark`** class, typically on `<html>` or a wrapper. Idiomatic for React/Vue with localStorage + className toggling.

The `color-mode` mixin in `foundation/_mixins.scss` emits both selectors in one rule, so the dark block costs one comma's worth of selectors and consumers pick whichever convention fits their stack. Consumers who need a different selector (e.g. `[data-color-mode="dark"]`) override at a higher `@layer`.

Token swap lives in `tokens/_theme.scss` as a single block:

```scss
:root {
  --st-background:       oklch(1    0 0);
  --st-foreground:       oklch(0.27 0 0);
  --st-surface:          oklch(1    0 0);
  --st-surface-2:        oklch(0.98 0 0);
  --st-border:           oklch(0.93 0 0);
  // ...
}

[data-theme="dark"],
.dark {
  --st-background:       oklch(0.155 0 0);
  --st-foreground:       oklch(0.97  0 0);
  --st-surface:          oklch(0.18  0 0);
  --st-surface-2:        oklch(0.21  0 0);
  --st-border:           oklch(0.27  0 0);
  // ...
}
```

Every component reads tokens, never literals. Dark mode is one block in one file.

**Asymmetry to remember.** `--st-warning-foreground` stays dark in both themes — yellow stays yellow regardless of mode, so the text on it stays dark. Other intent foregrounds flip to white in both modes; some components may invert via the dark block.

### 3.10 Site conventions (stisla.dev)

Patterns the site uses to demo components. Site-only, never bundled into the framework CSS or JS.

- **Per-component pages.** One `.njk` per component under `src/site/pages/<name>.njk`. Cover every variant and state.
- **Pages mirror the source tree, not the user mental model.** A page demos what its source file owns and nothing more. `/typography` demos typographic *patterns* (`.h1`–`.h6`, `.lead`, `.display-*`, `.prose`). Single-property text *utilities* (`.fw-*`, `.fs-*`, `.text-*`, `.lh-*`, `.text-{align,transform}-*`) live on a future `/text-utilities` page (or whatever the utilities entry is called), not on `/typography`. Same separation as `foundation/` vs `utilities/` in source. Site-page scope = source-file scope.
- **Customization section.** Every per-component page ends with a `<h2>Customization</h2>` section that tables the component's local `--component-*` vars in a `<div class="prose"><table>` (columns: Variable / Default / Use). Reference shape: `src/site/pages/slider.njk`. Source defaults from the component SCSS, never from memory. High-var components (e.g. `.btn` with 40) split into grouped sub-tables under `<h3>` headings. Components that reuse another's vars (`.toggle` reads `.btn`'s surface; `<select class="form-control">` reads form-control's) get a pointer section linking to the parent and table only the deltas or state-specific overrides.
- **Docs voice rules.** Demo-page prose stays plain: short `<p class="lead">` (just "what the component is"); no em dashes; no pixel-spec callouts in prose (say what the variant *is*, not its numeric values); fewer "X: Y" colon explanations; no AI-tells (`leverage`, `robust`, `seamless`, `delve`).
- **Demo viewer.** `src/site/partials/_demo.njk` exposes `{% call ui.demo() %}<body>{% endcall %}`. Framed preview on top, escaped source below. Single source of truth via Nunjucks `caller()`.
- **Dedent filter.** `tools/nunjucks-filters.mjs` strips common leading whitespace from demo bodies so code blocks render flush-left while `.njk` source stays indented.
- **Site JS.** `src/site/scripts/site.js` (built as `site-dist/assets/site.js`) holds demo-only behaviors via `data-demo-*` attribute conventions (`data-demo-loading`, `data-theme-toggle`, ...). Event-delegated, declarative. Framework JS bundle (`src/js/index.js`) stays clean.
- **Site stylesheet.** Site chrome lives in `src/site/styles/site.scss` and builds as a separate Vite entry. Linked from `base.njk` alongside the framework bundle. Never bundled into the framework.
- **Theme toggle.** `<button data-theme-toggle>` in the top nav flips `[data-theme]` on `<html>` and persists to `localStorage`. A tiny FOUC-prevention script in `<head>` applies the saved theme synchronously before paint.

### 3.11 Spec vs implementation

Stisla is a **design specification**. The spec defines what a component is — its anatomy, parts, states, behavior, a11y requirements, token usage, and visual identity. The spec is framework-agnostic.

Stisla has **implementations**. Each implementation realizes the spec idiomatically for its environment. v3 ships one implementation (vanilla). Others come later as separate packages.

**What lives in the spec (shared across every implementation):**
- The 28 `--st-*` tokens and what they mean
- BEM class names (`.btn`, `.modal`, `.dropdown-menu`, etc.) and the visual styling those classes produce
- State attribute conventions (`data-state="open"`, `data-orientation`, `data-disabled`, `data-highlighted`) — and the styling those attributes trigger
- Component anatomy (what parts a modal has, what states a dropdown supports)
- Behavior + a11y requirements (keyboard nav, focus management, ARIA roles)

**What lives per-implementation:**
- The JS API surface (`new Stisla.Modal(el)` in vanilla; `<Modal>` component in React)
- State management (vanilla toggles `data-state`; React's Radix Dialog manages it internally)
- Package shape and install instructions

**Three integration tiers in practice:**

| Consumer | Install | Author |
| --- | --- | --- |
| Plain HTML / Rails / Django / Laravel | `@stisla/css` + `@stisla/vanilla` | BEM classes + `data-stisla-*` declarative attrs; `Stisla.init()` |
| React | `@stisla/css` + `@radix-ui/react-*` (or `@base-ui/react`) | Radix/Base UI primitives + Stisla classes via `className` |
| Vue | `@stisla/css` + `reka-ui` | Reka UI primitives + Stisla classes |
| Svelte | `@stisla/css` + `bits-ui` (or `melt-ui`) | Bits primitives + Stisla classes |

The CSS layer is portable because it reads `[data-state="open"]` etc. — Radix/Base UI/Reka UI all expose these attributes natively. The same Stisla CSS file works in every environment without a glue layer.

**Why no Stisla React/Vue wrapper in 3.0.** Framework users can already adopt today (just install `@stisla/css` + their primitive library of choice). Wrappers would be convenience packages we ship later once we see what's worth wrapping. Out of scope for 3.0 (§6).
